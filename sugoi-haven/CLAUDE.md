# CLAUDE.md — Sugoi Haven project briefing

This file briefs Claude Code (or any fresh Claude session) on this project.
It encodes decisions made across the original design conversations. Read
README.md as well — it has operational detail this file doesn't repeat.

## What this is

An e-commerce site for ukiyo-e Japanese woodblock prints (Hokusai's
Thirty-Six Views of Mount Fuji, Hiroshige's Fifty-Three Stations of the
Tōkaidō). Live at https://sugoihaven.com (Netlify site "cheery-cactus-eac4da"), deploys
automatically from the `main` branch of this GitHub repo. The owner is
non-technical: explain changes plainly, avoid jargon in anything user-facing,
and never assume they'll debug something themselves.

## The inviolable data-model rule

Three tiers: **Series** (a shop set) → **Design** (canonical artwork, exactly
ONE page + ONE grid tile at `/shop/print/[slug]`, always) → **Listing** (one
physical copy with its own publisher, condition, price, photos).

NEVER create a new Design/page/tile for a publisher or condition variant —
those exist only as Listings compared side by side on the design's page.
The inventory validator enforces this by rejecting unknown design_ids.

## Data flow (do not hand-edit generated files)

- `src/data/designs.json` (55 Tōkaidō) — generated by `scripts/build-data.mjs`
  from the owner's seed CSV.
- `src/data/fuji-designs.json` (46 Hokusai) — generated by
  `scripts/build-fuji-data.mjs`.
- `src/data/listings.json` — generated by `scripts/fetch-inventory.mjs`,
  which runs as `prebuild`. Source of truth is the owner's **Google Sheet**
  (published CSV, `INVENTORY_CSV_URL` env var on Netlify). Locally with no
  env var it falls back to `scripts/demo-listings.json`. Validation is strict
  and fails builds loudly with Sheet row numbers — this is a feature, keep it.
- `src/data/catalog.js` — the only hand-edited data file (Series definitions,
  helpers, artists, condition meanings).

## Build modes (automatic, don't hand-toggle)

- `npm run build` → deploy build: BrowserRouter, clean URLs, then a
  `postbuild` prerender (`scripts/prerender.mjs`) renders ~122 routes to
  static HTML with per-page SEO tags, JSON-LD, sitemap.xml, robots.txt,
  and a real `dist/404.html`.
- `SINGLEFILE=1 npm run build` → one self-contained preview HTML
  (HashRouter); prerender auto-skips.
- There is deliberately NO SPA catch-all redirect (`netlify.toml`): every
  real route is a prerendered file, unknown URLs must 404 for SEO. Do not
  re-add a `/* -> /index.html 200` rule.

## SEO system

`src/lib/seo.js` is the single source of truth for titles/descriptions/
JSON-LD, consumed at build time by `scripts/prerender.mjs` and at runtime by
`src/lib/useSEO.js` (Layout). If you add a route, add it there too, or
prerender will skip it. Search/404 are noindex and excluded from the
sitemap on purpose (cart/checkout aren't routes at all — see Checkout
section below). `SITE_URL` env var switches the whole system to a custom
domain (already set to https://sugoihaven.com in production).

## Images

- Browse images are public-domain Wikimedia Commons reference scans,
  DEV PLACEHOLDERS ONLY, resolved by a candidate-fallback chain
  (`src/components/PrintImage.jsx`) with a localStorage cache
  (`sugoi-haven-img-cache-v1`) and an on-brand SVG placeholder floor.
- `scripts/verify-images.mjs` (run on a machine with open internet) bakes
  the winning filename into the data — encourage the owner to run it.
- Real Listing photos come from the Sheet's `photo_urls` column
  (pipe-separated URLs) and always override reference scans. Hard rule from
  the blueprint: a real listing must show photos of the exact physical sheet.

## Design system rules

Tokens in `src/index.css` (Tailwind v4 `@theme`): washi cream bg (never pure
white), sumi ink text, aizuri indigo primary, bokashi gradient transitions.
The warm `ume` accent is used ONLY for CTAs and prices — do not spread it.
Fonts: Noto Serif JP (display) + Noto Sans JP (UI). Decorative motifs are
original line art in `src/components/Marks.jsx` (never crop/copy the actual
artworks for UI chrome). GSAP animations are subtle (<0.8s), scroll-triggered
via `[data-reveal]` + `usePageReveals` — any NEW page using DesignCard or
`data-reveal` MUST call `usePageReveals()` or content renders invisible
(this bug happened once; see PrintPage).

## Testing

`node scripts/smoke.mjs` after `npm run build` — jsdom-mounts ~17 routes
against the deploy bundle (real paths, not hash URLs) and probes key content.
Run it before pushing anything nontrivial. Note its limitation: it checks
text presence, not visual rendering or opacity.

## Checkout — real, via Snipcart (not mock)

Cart + checkout is Snipcart (overlay script, no backend of ours). There are
NO `/cart` or `/checkout` routes anymore — deliberately removed. The old
`CartContext.jsx` and `CartCheckout.jsx` are deleted; don't recreate them.
The header's Cart button uses Snipcart's own `snipcart-checkout` /
`snipcart-items-count` classes.

**Critical pattern, don't "simplify" this away:** `PrintPage.jsx` renders one
`snipcart-add-item` button PER LISTING (not just the selected one), with only
the selected one visible via the `hidden` attribute. This is required because
Snipcart verifies every price by re-crawling the item's `data-item-url` page
looking for a matching `data-item-id`/`data-item-price` element — since the
page defaults to showing the cheapest listing, a customer buying a pricier
publisher's copy needs THAT listing's button to also exist in the page's real
(prerendered) HTML, not just the one visible at generation time. If you ever
touch this component, keep all listings' buttons in the DOM.

API key: `VITE_SNIPCART_API_KEY` env var (note the `VITE_` prefix — required,
unlike `SITE_URL`/`INVENTORY_CSV_URL`). Substituted into `index.html` via
Vite's built-in `%VAR%` HTML replacement. A small inline guard script skips
loading snipcart.js entirely if the placeholder never got replaced, so an
unconfigured deploy doesn't throw failed-auth errors on every pageview.

Snipcart's own stock tracking (via `data-item-max-quantity="1"`) is separate
from the Google Sheet — the owner needs to manually clear abandoned Snipcart
carts for anything they mark `sold` in the Sheet, the two systems don't talk
to each other.

Snipcart's cart/checkout UI is themed via CSS custom properties at the
bottom of `src/index.css` ("Snipcart theme override") — their officially
documented theming surface, mapped to the site's washi/aizuri/ume tokens.
Don't attempt deeper structural CSS overrides (e.g. to remove the empty
vertical space in a lightly-filled side-cart panel) — that's Snipcart's own
layout stretching to fill available height, fighting it risks breaking their
responsive behavior.

Snipcart's dashboard has its OWN separate domain allow-list (Store
Configuration → Domains & URLs) distinct from our `SITE_URL` env var — both
must say the real production domain or checkout fails with "URL of some
products could not be reached." This bit the owner once already; if it
recurs after a domain change, check both places.

**Fixed bug, don't reintroduce:** Snipcart's product custom fields
(Publisher/Condition/Print year) MUST carry `data-item-custom{N}-type="readonly"`
in `PrintPage.jsx`'s `SnipcartAddToCart`. Without it, Snipcart's default
behavior treats custom fields as CUSTOMER-EDITABLE inputs (its intended use
case is things like "engraving text" or "gift note") — meaning a customer
could literally type over "Yuyudo" / "Fine" with anything they want, and
that tampered text is what would show on the actual order. `readonly` is
what makes them plain display-only info instead.

**Fixed bug, don't reintroduce:** Snipcart's overlay (cart/checkout/the
post-order "Thank you" screen) lives outside React's tree — it doesn't know
when our router navigates. Without intervention, clicking a nav link while
it's open changes the URL and swaps the underlying page, but Snipcart's
overlay stays visually on top, making the site look frozen. Fixed in
`src/lib/useSEO.js`: calls `window.Snipcart.api.theme.cart.close()` on every
route change after the initial mount (guarded with a `mounted` ref so it
never fires on first page load, and with optional chaining + try/catch so it
never throws if Snipcart isn't loaded). If any other place in the app opens
a Snipcart button/link, this hook already covers it — no per-page fix needed.

## What is mock / not yet built

  localStorage (`sugoi-haven-waitlist-v1`) — must be pointed at a real
  provider before they're trusted.
- Remaining roadmap: Search Console/Bing submission (done as of this
  writing), owner's photography pass (ongoing), phase-2 "compare copies"
  toggle on print pages.

## Workflow expectations

Commits to `main` auto-deploy to production via Netlify. Before pushing:
build + smoke test locally. Explain every change to the owner in plain
language. Prefer small, reviewable commits. The owner updates inventory via
Google Sheet + a Netlify build-hook bookmark — never require them to edit
code or JSON for routine inventory work.
